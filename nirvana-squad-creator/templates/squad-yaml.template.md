# Squad Manifest Template (squad.yaml)

Gere um `squad.yaml` seguindo esta estrutura. Campos marcados `<!-- REQUIRED -->` sao obrigatorios. O exemplo usa um squad ficticio `etl-squad` como referencia visual.

## Exemplo de Referencia

```yaml
# --- Identity ---
name: etl-squad                    # <!-- REQUIRED --> kebab-case, unico entre squads
version: 1.0.0                     # <!-- REQUIRED --> semantic versioning (MAJOR.MINOR.PATCH)
description: "ETL pipeline squad for data warehouse ingestion"  # <!-- REQUIRED --> texto conciso

author: "Generated by Nirvana Squad Creator"   # Nome e/ou email do autor
license: MIT                                    # Identificador de licenca (MIT, Apache-2.0, etc.)
slashPrefix: etl                                # Prefixo de comandos no IDE (habilita /etl:comando)

# --- AIOS Compatibility ---
aios:                              # <!-- REQUIRED --> bloco de compatibilidade
  minVersion: "2.1.0"             # <!-- REQUIRED --> semver, versao minima do AIOS Core
  type: squad                      # <!-- REQUIRED --> sempre "squad"

# --- Components ---
components:                        # <!-- REQUIRED --> lista de arquivos por tipo
  agents:                          # <!-- REQUIRED --> lista de agents .md
    - data-extractor.md            # Um arquivo por agente
    - transformer.md
    - loader.md
    - qa-validator.md
  tasks:                           # <!-- REQUIRED --> lista de tasks .md
    - extract-data.md
    - transform-records.md
    - load-to-warehouse.md
    - validate-pipeline.md
  workflows:                       # <!-- REQUIRED --> lista de workflows .yaml
    - etl-pipeline.yaml
    - data-validation.yaml
  checklists:                      # Checklists de validacao (opcional)
    - pre-extraction-checks.md
  templates:                       # Templates de documentos (opcional)
    - extraction-report.md
  tools: []                        # Ferramentas customizadas (opcional)
  scripts:                         # Scripts utilitarios (opcional)
    - scripts/validate-source.sh
    - scripts/check-schema.sh

# --- Config Inheritance ---
config:
  extends: extend                  # extend | override | none
  coding-standards: config/coding-standards.md   # Regras de estilo de codigo
  tech-stack: config/tech-stack.md               # Tecnologias e ferramentas
  source-tree: config/source-tree.md             # Estrutura de diretorios

# --- Dependencies ---
dependencies:
  node:                            # Pacotes npm
    - "pg@^8.11.0"
    - "csv-parser@^3.0.0"
  python: []                       # Pacotes pip
  squads:                          # Outros squads (formato semver: "name@^version")
    - "monitoring-squad@^1.0.0"

# --- Discovery ---
tags:                              # Tags para busca e categorizacao
  - etl
  - data-pipeline
  - warehouse
  - ingestion
```

Schema completo com todos os campos, tipos, e restricoes: [references/squad-yaml-schema.md](../references/squad-yaml-schema.md)

## Validation Rules

1. `name` em kebab-case (letras minusculas, numeros e hifens)
2. `version` e `aios.minVersion` em semver valido (MAJOR.MINOR.PATCH)
3. `aios.type` sempre `"squad"`
4. Todos os arquivos listados em `components` existem no diretorio do squad
5. `config.extends` e um dos valores: `extend`, `override`, `none`
6. **`description` DEVE ser string inline entre aspas** — NUNCA usar YAML multi-line (`|` ou `>`). O parser AIOS não valida blocos multi-line e interpreta como `object,object` em vez de string. Sempre usar: `description: "texto conciso em uma linha"`
